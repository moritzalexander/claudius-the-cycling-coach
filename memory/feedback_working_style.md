Do not build multiple things without discussing each one first. Walk through the purpose, logic, sources, and trade-offs of each skill/component before implementing.

**Why:** Moritz explicitly called out that skills were built without being reviewed and discussed individually as requested. He wants to understand and shape each piece, not just receive finished output.

**How to apply:**
- Present the design for each skill/component and wait for feedback before writing code
- When building multiple skills, go one at a time — discuss → agree → build → next
- Ask questions when there are trade-offs rather than making assumptions
- The user values reuse and pragmatism over originality — always check what exists first
- The user is comfortable with up to 5 interactive questions in a coaching flow
- The user wants explicit categorization shown to them (not implicit)
